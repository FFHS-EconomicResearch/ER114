1 Lernziele und Überblick

HinweisLernziele dieses Labs
  • Das multiple Regressionsmodell als Scoring-Instrument einsetzen
  • Residuen als quantitatives Underperformance-Maß berechnen und ranken
  • Ergebnisse im Managerial Memo managementtauglich kommunizieren (Translator-Rolle)
  • Die Grenzen des Modells kritisch reflektieren: R², OVB, Data Drought, Make-or-Buy

Szenario: Die Live-Demo hat gezeigt, dass das multiple Modell Sales ~ Price + Advertising + ShelveLoc rund 54% der Umsatzvarianz erklärt. Jetzt nutzen wir die Residuen als Performance-Score und erstellen die Underperformer-Liste — das ist CRISP-DM Phase 5 (Evaluation) und Phase 6 (Deployment).

2 Hintergrundmaterial: Vertiefung multiple Regression

Dieses Kapitel enthält ergänzendes Hintergrundmaterial, das Sie vor oder nach dem Lab lesen können. Es vertieft die Konzepte aus der Live-Demo anhand weiterer Datensätze.

  • Simpson’s Paradoxon
  • Kategoriale Regression (Gapminder)
  • Multiple Regression: 2 numerische Regressoren

Das Problem: Ein aggregierter Trend kann dem Trend innerhalb von Subgruppen entgegengesetzt sein — wenn ein Confounder beide beeinflusst.

Im Carseats-Datensatz: Filialen mit gutem Regalplatz (Good) können höhere Preise verlangen und haben mehr Umsatz. Das überlagert den negativen Preiseffekt im einfachen Modell.

3.2 Schritt 2 — Performance-Scores berechnen

filial_scored <- mod_multipel %>%
  get_regression_points() %>%
  mutate(
    performance_gap = residual,
    performance_kat = case_when(
      residual < -2 ~ "Underperformer",
      residual >  2 ~ "Outperformer",
      TRUE          ~ "Normal"
    )
  )

# Überblick
filial_scored %>%
  count(performance_kat) %>%
  mutate(anteil = scales::percent(n / sum(n), accuracy = 0.1)) %>%
  arrange(desc(n)) %>%
  gt() %>%
  tab_options(table.font.size = "13px")
performance_kat n anteil
Normal 293 73.2%
Outperformer 56 14.0%
Underperformer 51 12.8%
HinweisResiduum = Underperformance-Maß

\[e_i = \text{Sales}_i - \widehat{\text{Sales}}_i\]

  • \(e_i \ll 0\): Filiale verkauft deutlich weniger als der Benchmark — Underperformer
  • \(e_i \approx 0\): Filiale liegt im Marktdurchschnitt
  • \(e_i \gg 0\): Filiale verkauft mehr als erwartet — Outperformer

Wichtig: Der Benchmark ist bereits für Preis, Werbung und Regalplatz bereinigt — ein Underperformer ist also wirklich schlechter als Filialen mit vergleichbaren Ausgangsbedingungen.

4 Gruppenaufgabe: Managerial Memo

Vorsicht Gruppenarbeit (3–4 Personen, 15 Minuten)

4.1 Auftrag

Schreiben Sie ein kurzes Memo an den Geschäftsführer (max. 300 Wörter). Sie sind das Data-Science-Team — der GF ist kein Statistiker.

Beantworten Sie alle drei Fragen klar, präzise und managementtauglich.

4.2 Frage 1 — Modell-Evaluation: Ist R² ausreichend?

„Unser Modell erklärt R² ≈ 54% der Umsatzvarianz. Empfehlen Sie dem Management, auf Basis dieses Modells Filialen zu schließen? Begründen Sie Ihre Empfehlung.”

Was macht eine starke Antwort aus?

Schwache Antwort Starke Antwort
„R² = 0.54, das ist über 50%, also gut genug.” „R² = 0.54 bedeutet: 46% der Umsatzvarianz ist durch unbekannte Faktoren bedingt. Eine Schließungsentscheidung auf dieser Basis birgt erhebliches Fehlerrisiko — wir empfehlen zunächst eine gezielte Datenergänzung, bevor irreversible Maßnahmen ergriffen werden.”

4.3 Frage 2 — OVB: Welche Variablen fehlen?

„Welche Faktoren fehlen in unseren Daten, die erklären könnten, warum eine Filiale underperformt? Nennen Sie drei konkrete Beispiele mit Begründung.”

Was macht eine starke Antwort aus?

Schwache Antwort Starke Antwort
„Es fehlen mehr Daten über die Kunden.” „Bevölkerungsdichte im 5-km-Radius: Filialen in dünn besiedelten Regionen haben strukturell niedrigere Umsätze — unabhängig von Preis, Werbung und Regalplatz. Dieses Merkmal fehlt und verzerrt den Score für Landfilialen systematisch nach unten.”

4.4 Frage 3 — Data Strategy: Make or Buy?

„Sollen wir externe Geodaten und Demografiedaten von einem Datenhändler einkaufen (Commodity Data) oder eine eigene Kundenbefragung aufbauen (Proprietary Data)? Argumentieren Sie mit dem AI-Value-Creator-Konzept aus Thomas et al. (2025).”

Was macht eine starke Antwort aus?

Schwache Antwort Starke Antwort
„Wir sollten Daten kaufen — das ist schneller und günstiger.” „Geodaten (Commodity Data) sind für alle Wettbewerber beschaffbar — kein nachhaltiger Vorteil. Eine eigene Loyalty-Card mit Kaufhistorie (Proprietary Data) schafft einen Data Moat. Empfehlung: Kurzfristig Commodity kaufen (Pilotprojekt), langfristig Loyalty-Programm aufbauen für dauerhaften Wettbewerbsvorteil (Thomas et al., 2025: AI Value Creator).”

5 Debrief: Make-or-Buy und Expected Value Framework

5.1 Wann lohnt sich Dateneinkauf?

\[\underbrace{E[\Delta\text{Gewinn}]}_{\text{Mehrwert besserer Entscheidungen}} > \underbrace{C_{\text{Daten}}}_{\text{Beschaffungskosten}}\]

Parameter Wert
Kosten Demografiedaten (Commodity) 50.000 EUR/Jahr
Kosten falsche Filialschließung 500.000 EUR
P(Fehlentscheidung ohne Daten) 30%
P(Fehlentscheidung mit Daten) 10%
E[Δ Gewinn] (0.30 − 0.10) × 500.000 = **100.000 EUR**
Entscheidung ✅ Datenkauf lohnt sich — 100.000 > 50.000
HinweisVom AI User zum AI Value Creator

Das Pilotprojekt (heute) hat gezeigt: Commodity Data (öffentlicher Carseats-Datensatz) reicht für einen Proof-of-Concept — aber nicht für nachhaltige Wettbewerbsvorteile.

Der Weg zum AI Value Creator (Thomas et al., 2025):

  1. Kurzfristig: Externe Geodaten einkaufen (Commodity) → Modell verbessern
  2. Mittelfristig: Loyalty-Programm aufbauen → eigene Transaktionsdaten
  3. Langfristig: Modell mit Proprietary Data trainieren → nicht kopierbar

### Simpson's Paradoxon

**Das Problem:** Ein aggregierter Trend kann dem Trend innerhalb von
Subgruppen entgegengesetzt sein — wenn ein Confounder beide beeinflusst.

Im Carseats-Datensatz: Filialen mit gutem Regalplatz (Good) können höhere Preise verlangen
**und** haben mehr Umsatz. Das überlagert den negativen Preiseffekt im einfachen Modell.

```{r simpson-viz}
p_agg <- tbl_carseats %>%
  ggplot(aes(x = Price, y = Sales)) +
  geom_point(alpha = 0.3, color = "#686868", size = 1.5) +
  geom_smooth(method = "lm", color = "#D50006",
              se = FALSE, linewidth = 1.3) +
  annotate("text", x = 70, y = 16,
           label = paste0("Gesamt: r = ",
                          round(cor(tbl_carseats$Price,
                                    tbl_carseats$Sales), 3)),
           color = "#D50006", size = 3.5, hjust = 0) +
  labs(title = "Aggregiert (ohne Gruppen)",
       x = "Preis (USD)", y = "Sales (Tsd.)") +
  theme_minimal(base_size = 10)

p_grp <- tbl_carseats %>%
  ggplot(aes(x = Price, y = Sales, color = ShelveLoc)) +
  geom_point(alpha = 0.35, size = 1.3) +
  geom_smooth(method = "lm", se = FALSE, linewidth = 1.1) +
  scale_color_manual(values = c("Bad"    = "#D50006",
                                "Medium" = "#686868",
                                "Good"   = "#502479")) +
  labs(title = "Nach Regalplatz (ShelveLoc)",
       subtitle = "Innerhalb jeder Gruppe: gleicher negativer Trend",
       x = "Preis (USD)", y = "Sales (Tsd.)",

  )
```

**Takeaway:** Immer nach Confoundern suchen und Subgruppenanalysen durchführen,
bevor Sie Aggregate interpretieren. Das multiple Modell löst dieses Problem,

```{r scores}
filial_scored <- mod_multipel %>%
  get_regression_points() %>%

    performance_gap = residual,

      residual < -2 ~ "Underperformer",
      residual >  2 ~ "Outperformer",
      TRUE          ~ "Normal"
    )
  )
```

:::{.callout-note}
## Residuum = Underperformance-Maß

$$e_i = \text{Sales}_i - \widehat{\text{Sales}}_i$$

- $e_i \ll 0$: Filiale verkauft **deutlich weniger** als der Benchmark — **Underperformer**
- $e_i \approx 0$: Filiale liegt im Marktdurchschnitt
- $e_i \gg 0$: Filiale verkauft **mehr** als erwartet — **Outperformer**

**Wichtig:** Der Benchmark ist bereits für Preis, Werbung und Regalplatz
*bereinigt* — ein Underperformer ist also wirklich schlechter als
Filialen mit vergleichbaren Ausgangsbedingungen.
:::

## Debrief: Make-or-Buy und Expected Value Framework

### Wann lohnt sich Dateneinkauf?

$$\underbrace{E[\Delta\text{Gewinn}]}_{\text{Mehrwert besserer Entscheidungen}} > \underbrace{C_{\text{Daten}}}_{\text{Beschaffungskosten}}$$

```{r ev-table}
#| echo: false
tibble(
  Parameter = c(
    "Kosten Demografiedaten (Commodity)",
    "Kosten falsche Filialschließung",
    "P(Fehlentscheidung ohne Daten)",
    "P(Fehlentscheidung mit Daten)",
    "E[Δ Gewinn]",
    "Entscheidung"
  ),
  Wert = c(
    "50.000 EUR/Jahr",
    "500.000 EUR",
    "30%",
    "10%",
    "(0.30 − 0.10) × 500.000 = **100.000 EUR**",
    "✅ Datenkauf lohnt sich — 100.000 > 50.000"
  )
) %>%
  gt() %>%
  tab_style(
    style     = cell_fill(color = "#f0fff0"),
    locations = cells_body(rows = Parameter == "Entscheidung")
  ) %>%
  tab_style(
    style     = cell_fill(color = "#fff8e1"),
    locations = cells_body(rows = Parameter == "E[Δ Gewinn]")
  ) %>%
  tab_options(table.font.size = "13px")
```

:::{.callout-note}
## Vom AI User zum AI Value Creator

Das Pilotprojekt (heute) hat gezeigt: **Commodity Data** (öffentlicher Carseats-Datensatz)
reicht für einen Proof-of-Concept — aber nicht für nachhaltige Wettbewerbsvorteile.

Der Weg zum **AI Value Creator** (Thomas et al., 2025):

1. **Kurzfristig:** Externe Geodaten einkaufen (Commodity) → Modell verbessern
2. **Mittelfristig:** Loyalty-Programm aufbauen → eigene Transaktionsdaten
3. **Langfristig:** Modell mit Proprietary Data trainieren → nicht kopierbar

:::

1. **Residuum = Performance-Score:** Das multiple Modell liefert einen preis-,

## Ausblick: Was kommt in PVA2?

Heute haben wir **deskriptiv** gearbeitet: alle 400 Filialen = Grundgesamtheit (Population).

**Offene Frage:** Was wenn die 400 Filialen nur eine **Stichprobe** wären?
Wie sicher sind unsere Koeffizienten? Hätten wir bei einer anderen Saison
dieselben Ergebnisse? Und wie wahrscheinlich ist es, dass der gemessene Werbeeffekt
nur Zufall ist?

Das sind die Fragen der **statistischen Inferenz** — das Thema von PVA2:
Bootstrap, Konfidenzintervalle, p-Hacking, FDR und das Expected-Value-Framework
für Manager.
:::
